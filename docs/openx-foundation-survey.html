一、我们要找的地基是什么

OpenX 不是「再做一个会干活的 Agent」，而是工头层：用户只关心目标与效果，执行黑盒。

MVP 已定：本机 PC · Goal 五态 · 用户立目标 + 效果确认 · 单 Executor · 模板播报。

按层拆：哪些可以借，哪些必须自研

层需要什么借地基的难度
L1 目标看板 UI列表、筛选、详情、达标/返工中等 — several 有 Kanban，但心智多偏「Agent 自治」
L2 任务/状态服务Goal CRUD、五态、SQLite、SSE容易 — 多个 TS 项目同款栈
L3 执行桥调一个本机 Executor，不回暴露 CLI 品牌中等 — ACP 最接近「统一接 CLI」
L4 执行/推理写代码、工具循环、沙箱不借 — OpenHands / LangGraph 属此层，会跑偏
L5 协议Agent 互操作、流式 UI可选 — A2A / AG-UI 作远期，MVP 不必全上
结论：找地基 = 找 L1+L2 的可复用壳，L3 用协议或薄适配；不要把整个产品建在 OpenHands / LangGraph 之上。

关键差异说明

  • Agent Kanban：有「In Review → 人 reject/complete」，和你们的效果确认很像；但官方 UI 不让人创建任务（任务由 Agent/CLI 创建），和「用户立目标」相反。多 runtime（Codex/Cursor…）外露，和「用户不选工具」相反。技术栈可参考：React + Hono + D1 + SSE。
  • Mission Control：SQLite、本机、Kanban、多 Agent — 像「加强版指挥台」，但功能面过宽（32 面板、多框架适配），MVP 易被拖进编排框架坑。
  • Agent Cockpit：本机、审批、时间线很强，心智是「并行 coding 会话」，不是「目标达成」；适合抄审批/日志 UX，不适合整仓 fork。
  • LangGraph / OpenHands：会把你们变成「又一个执行型 Agent 平台」，和工头定位冲突。
  • ACP：不解决看板，但适合当唯一 Executor 桥：对用户只显示「执行位」，背后接任意 ACP Agent。

三、建议：怎么借，不整仓 fork 谁

✅ 推荐路线（薄地基 + 自研工头）

主策略
  • 自研 Goal 模型、五态、看板 UI（沿用 stage-design 原型）
  • 借鉴实现：Agent Kanban 的 SSE 推送、任务生命周期、review 闸门（改为人可创建 Goal）
  • 借鉴栈：Mission Control 的 SQLite + 单进程本机部署思路
  • 执行桥：本机 Executor 进程；远期用 ACP 接 Claude/Codex/OpenCode，对用户仍是一个「执行位」

⚠️ 可参考、不当作根仓库

零件库
  • Agent Cockpit：审批 inbox、时间线、Diff 折叠（放进 Goal 详情）
  • AG-UI：若前端要标准化流式事件，可只借事件形态，不必上 CopilotKit 全家桶
  • A2A：等多 Executor / 远程再考虑，MVP 过重

❌ 不建议作为主地基

会跑偏
  • LangGraph / CrewAI / AutoGen — 执行编排框架
  • OpenHands 全栈 — 沙箱 + 写代码 Agent
  • 整仓 fork Agent Kanban — 产品哲学冲突（Agent-first vs 人立目标）
一句话：市面没有现成「只做工头、用户立目标、隐藏 CLI」的成品；最接近的是 Agent Kanban / Mission Control 的看板 + 状态机 + 本机栈，但必须砍掉多 runtime 外露和 Agent 自治建任务，并加上你们的 Goal/验收/效果确认。

四、落在 OpenX 上的推荐组合（MVP）

┌─────────────────────────────────────────┐ │ OpenX Web（自研，机械朋克 UI） │ │ 目标看板 · 详情 · 达标/返工 · 播报 │ └──────────────────┬──────────────────────┘ │ REST + SSE（自研，可参考 Kanban） ┌──────────────────▼──────────────────────┐ │ OpenX Server（自研） │ │ Hono/Fastify + SQLite + Goal 五态 │ │ 工头播报（模板，无 LLM 也可） │ └──────────────────┬──────────────────────┘ │ 内部 API（你们已定 v0.1） ┌──────────────────▼──────────────────────┐ │ Executor（薄层，可替换） │ │ MVP: mock 或 单 CLI 子进程 │ │ V2: ACP Client → 任意注册表 Agent（用户不可见品牌） │ └─────────────────────────────────────────┘ 不引入：LangGraph 图、OpenHands Docker 沙箱、Mission Control 32 面板

为何 ACP 值得单独作为「执行桥协议」

  • 定位是 Client(你们) ↔ Agent(CLI)，不是 Agent 互抢任务 — 和「单 Executor 黑盒」一致。
  • 已覆盖 Claude Code、Codex、Gemini、Copilot、OpenCode 等（经适配器），满足「背后可换、面前不换」。
  • 与 MCP 分工清晰：ACP 连 Agent，MCP 连工具 — 工头层不必碰 MCP。
  • 代价：需实现一个 ACP Client 壳（或嵌 Zed 式 subprocess 管理），工作量在一两个 Sprint，小于维护 N 套 CLI 适配器。

五、建议的验证步骤（1–2 周）

顺序动作产出
1 本地 clone Agent Kanban，只读走一遍：任务状态、SSE、review API 列「可抄清单」vs「要改清单」
2 浏览 Mission Control README + 跑起来看 Kanban 与 SQLite 结构 确认本机部署与表设计是否过重
3 读 ACP overview + 用一个官方 sample 连一个 CLI 验证 Executor 桥是否 2–3 天能通
4 在 OpenX monorepo 起 apps/server + apps/web，不 fork 上述仓库，只抄协议与表意 P0 可演示闭环

决策检查（拍板用）

  • 主地基 = 自研工头 + SQLite，不是 fork OpenHands/LangGraph。
  • 执行 = 单 Executor；协议储备 = ACP（不是 A2A 优先）。
  • 参考实现 #1 = Agent Kanban（生命周期 + review）；#2 = Mission Control（本机全家桶）。
