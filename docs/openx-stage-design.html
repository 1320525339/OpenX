定位 原理 模型 流程 交互原型 接口 阶段 边界

一、产品定位

OpenX 是「工头 + 统一目标看板」，不是又一个会写代码、会控电脑的 Agent 工具。

用户只思考：要达成什么、怎样算做好、整体进展如何。不思考该用 Codex、Cursor 还是 Claude。

市面上不缺执行型 Agent；缺的是能整体规划、汇总状态、推动效果确认的那一层。

我们做

  • 目标 → 执行 → 效果确认 闭环
  • 统一看板 + 工头播报
  • 单执行面（黑盒），本机 PC

不做

  • 多 CLI 品牌编排
  • 自建写代码 / 控桌面能力
  • 自由 Chat、小组 CRUD、任务 DAG

二、第一性原理（功能裁剪依据）

① 用户唯一输入 要达成什么 + 一句验收标准
② 用户唯一输出 看全盘推进 + 对结果说达标 / 返工
③ 系统唯一职责 存目标、调执行面、收结果、促确认
④ 执行不可见 子代理、CLI 品牌不对用户暴露

三、最小数据模型

只存 Goal（目标），不另建 Squad / Task 多套实体。

Goal {
  id, title, acceptance,    // 目标 + 怎样算做好
  status, progress,         // 五态 + 0-100
  assignee: "default",      // MVP 固定单一执行位
  resultSummary?, effectStatus?,  // 交差摘要 / approved | rework
  createdAt, updatedAt
}

状态机（5 态）— 点击节点查看说明

draft → running → awaiting_review → done | failed
running：执行面正在处理；进度由 Executor 回调更新。

达标 → done；返工 → 再 dispatch
B
看全盘
首页三筛选：进行中 | 待确认 | 已完成
C
详情
验收标准、结果、折叠技术日志

路径 A · 步骤 1
用户点击「新目标」，填写必填项：标题（一句话目标）、验收标准（怎样算做好）。点击「开始」后，OpenX 向本机 Executor 发送 POST /execute，Goal 进入 running。
